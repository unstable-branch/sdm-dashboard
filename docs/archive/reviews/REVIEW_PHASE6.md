## Provenance Verdict

**Verdict: PARTIAL — good baseline, with gaps.**

**What's recorded (in `manifest.json`):**
- Git SHA of deployed code
- R version + platform
- Package versions (from `sessionInfo()`)
- Occurrence file MD5 hash
- Cleaning summary (removed count, columns)
- Covariate names, source, file count
- CV strategy, k, threshold
- All model parameters (from config)
- Output file paths
- Run timestamp

**What's missing:**
- **Occurrence file SHA256** — manifest uses MD5 (`tools::md5sum`), not SHA256. The Plumber manifest (`plumber.R:432-436`) uses SHA256 but the `manifest.R` standalone version uses MD5.
- **Input covariate file hashes** — only file names are recorded, not hashes
- **Environment variable values** (non-secret) — not captured
- **CPU time / peak memory** — recorded in Plumber meta.json but not in the R manifest
- **Docker image SHA** — not recorded (relevant for containerised deployments)

**Plumber manifest (plumber.R:420-513) is more thorough** than the standalone `manifest.R`. It includes occurrence SHA256, resource tracking, and explicit parameter serialisation. This dual-manifest approach could cause confusion — two files, different schemas, different completeness levels.

## ODMAP Report

**File:** `R/output/report_odmap.R` (429 lines)

The ODMAP report follows Zurell et al. 2020 (cited). It populates:
- Overview: species, authorship, model type
- Data: occurrence source, cleaning, covariates (names, resolution)
- Model: algorithm, settings, CV strategy, thresholds
- Assessment: AUC, TSS, sensitivity, specificity, CBI
- Prediction: file paths, map extent

**Quality:** The ODMAP report is thoroughly populated from run metadata. It is NOT a template with placeholders — it pulls from the `result` object. ~90% of fields are auto-populated. User-specific fields (author name, hypotheses) use defaults (e.g., `"Anonymous"` for author).

## Nodata Handling

- Rasters: `COMPRESS=LZW`, `TILED=YES`. NODATA is NOT explicitly set in all cases:
  - `plumber.R:712`: `"NODATA=-9999"` for EPSG:3857 COGs
  - `predict_suitability` in `prediction.R` uses `terra::predict(na.rm=TRUE)` — NA cells return NA
  - `summarise_suitability` in `prediction.R:50` uses `suitability >= threshold` directly without explicit nodata handling
