# v2.0.0-beta.4 candidate evidence

This prerelease is a stabilization and release-discipline checkpoint for the v2 platform. It does not declare the deployment contract stable yet.

## Main corrections

- Fresh databases and named volumes initialize without manual SQL, ownership changes, or container restarts.
- Plumber parses PostgreSQL URLs into explicit RPostgres connection parameters.
- Completed artifacts, reports, diagnostics, provenance, and output metadata persist through the API synchronization path.
- WorldClim and CHELSA use persistent cache volumes, report per-file byte progress, retain resumable partial files, and survive container recreation.
- Projection statistics, threshold-area calculations, ODMAP resolution, and DNN diagnostics use real run data instead of misleading fallback arithmetic.
- Results navigation, mobile navigation, provenance counts, report discovery, and unavailable batch controls render truthfully.
- CPU, CUDA, and ROCm Plumber images have distinct runtime contracts and immutable release names.

## Release and deployment changes

- Tags drive five immutable GHCR artifacts: CPU/CUDA/ROCm Plumber, API, and frontend.
- Production application and external-service images are digest-pinned; application services never build in production Compose.
- Docker build bases and GitHub Actions are pinned, and the release audit rejects mutable inputs.
- Release publication produces source and Windows-ready archives, SHA256SUMS, image-digests.txt, SBOM/provenance, and a review-only draft release.
- Release tags must identify version-matching commits already contained in `main`.

## Local acceptance evidence

- Fresh empty-volume startup: PostgreSQL, Redis, Garage, Plumber, API, and frontend healthy.
- Product browser smoke: registration, authenticated startup, primary routes, and mobile drawer behavior passed.
- Fresh GLM/WorldClim workflow completed with expected TIFF, report, RDS, ODMAP, diagnostics, and provenance artifacts.
- Upgrade rehearsal preserved 2 users, 2 projects, 3 completed runs, and 19,483 output files.
- Rollback rehearsal restored the prior images with the same database records and files readable.
- CHELSA downloaded six selected global BIO layers (about 2.0 GB), survived Plumber recreation with identical SHA-256 checksums, and reused all six cached files.
- ROCm PyTorch DNN fit and prediction both reported `device: rocm`; the cached CHELSA rerun completed in 25 seconds.
- Final CPU image built at 3.32 GB and passed torch 0.17.0, cito 1.1, CPU tensor, DNN source, native extension, entrypoint, and fresh-volume write checks.
- Final CUDA image built at 8.71 GB on the NVIDIA host. Four mandatory native extensions compiled and linked cleanly; CUDA tensors, matrix operations, both DNN sources, fresh-volume permissions, and a real three-epoch cito DNN passed on two visible RTX 5060 Ti GPUs.
- The unfinished pinned-memory optimization is disabled by default behind the tested standard tensor-transfer fallback; opting into it remains a developer-only experiment.
- Node, R, Compose, accelerator, workflow-YAML, release-audit, conflict-marker, diff-integrity, and clean-worktree gates passed before release preparation.

## Schema, storage, and rollback

Database changes are additive. Back up PostgreSQL, object storage, and generated outputs before upgrading. Rollback was demonstrated by restoring the previous application images and restarting without rebuilding; the representative database and output volume remained readable.

Climate caches now live in named volumes mounted at `/app/Worldclim`, `/app/chelsa`, `/app/covariates`, and `/app/Worldclim_future`. Removing those volumes intentionally discards cached environmental data and causes later downloads.

## Known limitations before stable v2

- The published-digest production install can only be exercised after the draft release workflow has pushed candidate images.
- Stable `v2.0.0` still requires a post-publication soak and an installation from the published beta.4 digests.
